1. What is CoHabit-AI?


CoHabit-AI is a web application for college hostel administrators. It automatically assigns students to rooms based on personality compatibility — not randomly.


The Problem

  • Random allocation = roommate conflicts
  • Manual allocation = admin spends hours

    • The Solution

  • Stage 1 — AI Interview: Google Gemini chatbot interviews each student
  • Stage 2 — Smart Allocation: OR-Tools optimizer groups compatible students


    • 2. The Big Picture (7-Step Flow)


    Step 1: Create Session → define room inventory (e.g. 20 double rooms)

    Step 2: Enroll Students → add name, roll no, branch, gender

    Step 3: AI Interviews → Gemini asks lifestyle questions, stores conversation

    Step 4: Trait Extraction → Gemini reads transcript → outputs structured JSON

    Step 5: Compatibility Scoring → ML scorer gives 0-100 score to every pair

    Step 6: Room Allocation → OR-Tools finds globally optimal assignment

    Step 7: View Results → Admin sees room cards with members and scores

    5. The Database — All 8 Tables Explained


    Entity Relationships

    college (1) → (∞) allocation_sessions (1) → (∞) student

    |

    ├→ (1) interview (conversation JSON)

    └→ (1) traits (personality profile)

    allocation_sessions → (∞) compatibility_score (pairwise scores)

    allocation_sessions → (∞) recommendations → (∞) recommendation_members



    Table: college

    id → auto-increment primary key

    name → "IIT Bombay"

    email → login credential, UNIQUE

    password → bcrypt hash (NEVER plain text)


    WHY: Multi-tenant — each college sees only its own data.



    Table: allocation_sessions

    id → primary key

    college_id → FK to college

    title → "Batch 2025 First Year Hostel"

    academic_year → "2025-26"

    session_size → total students expected

    room_inventory → JSONB: {"2": 20, "3": 10} = 20 double rooms + 10 triple rooms

    session_status → "Draft" | "Active" | "Completed"


    WHY JSONB: Any mix of room sizes without schema changes. PostgreSQL JSONB is stored in binary for fast querying.



    Table: student

    id → primary key

    college_id → FK to college

    allocation_session_id → FK to allocation_sessions

    name → "Ravi Kumar"

    roll_number → "CS2024001" — UNIQUE

    email → UNIQUE

    branch → "Computer Science"

    year_of_study → 1–4

    gender → "Male" | "Female" | "Other" ← HARD CONSTRAINT (no mixed-gender rooms)



    Table: interview

    id → primary key

    student_id → FK to student

    conversation → TEXT containing JSON array:

    [{"role":"model","text":"Hi! Please answer..."},

    {"role":"user","text":"I sleep at 11pm..."}]

    completed_at → timestamp


    WHY TEXT not a separate messages table: Gemini API returns a list. We serialize it once

    with json.dumps(), store it, and json.loads() when needed. Avoids complex joins.



    Table: traits (Personality Profile)

    student_id → UNIQUE FK to student (one profile per student)

    sleep_time → "23:00"

    wake_time → "06:30"

    study_style → "alone in silence"

    noise_tolerance → FLOAT 0.0–1.0 (0=hates noise, 1=fully tolerant)

    cleanliness → FLOAT 0.0–1.0 (0=messy, 1=extremely neat)

    social_level → FLOAT 0.0–1.0 (0=extreme introvert, 1=extrovert)

    preferred_room_size → INT 1–4 (HARD CONSTRAINT for optimizer)

    flexible_preferences → "Can compromise on noise level"

    non_negotiable_preferences → "Must have quiet study time after 10pm"

    personality_summary → AI-written 2-3 sentence description


    WHY 0–1 floats: Normalization makes subtraction meaningful. |0.9 – 0.2| = 0.7 = very different.



    Table: compatibility_score

    allocation_session_id → FK to session

    student1_id → FK to student

    student2_id → FK to student

    compatibility_score → DECIMAL 0.00–100.00


    WHY PRECOMPUTED: N*(N-1)/2 pairs. For 20 students = 190 pairs. Computed once,

    read many times by the optimizer. Classic performance pattern.



    Table: recommendations

    id → primary key

    allocation_session_id → FK to session

    room_number → "2-Sharing Room 1" (format encodes capacity)

    compatibility_score → average score of all pairs in this room

    reason → explanation text



    Table: recommendation_members (Junction Table)

    recommendation_id → FK to recommendations

    student_id → FK to student

    UNIQUE(recommendation_id, student_id)


    WHY SEPARATE: Many-to-many (one room has many students, one student in one room).

    Junction table is the correct relational database pattern.

    ml/scorer.py — Compatibility Score 0-100


    def calculate_compatibility(traits1, traits2):

    score = 100.0 # start perfect


    # Each mismatch deducts points:

    noise_diff = abs(traits1["noise_tolerance"] - traits2["noise_tolerance"])

    score -= noise_diff * 20.0 # noise: 20 pts


    clean_diff = abs(traits1["cleanliness"] - traits2["cleanliness"])

    score -= clean_diff * 20.0 # cleanliness: 20 pts


    social_diff = abs(traits1["social_level"] - traits2["social_level"])

    score -= social_diff * 10.0 # social: 10 pts


    # Time comparison with normalization (handles "10pm" == "22:00")

    score -= time_penalty(traits1["sleep_time"], traits2["sleep_time"], max=15.0)

    score -= time_penalty(traits1["wake_time"], traits2["wake_time"], max=15.0)


    # Study style keyword clustering

    # "silent" vs "noisy" = full 20pt penalty

    # Same cluster = 0 penalty

    score -= study_style_penalty(...)


    return max(0.0, min(100.0, score)) # clamp


    WEIGHTS: Noise=20, Cleanliness=20, Study style=20, Sleep=15, Wake=15, Social=10 (total=100)

    TIME NORMALIZATION: "10pm" → 1320 min, "22:00" → 1320 min. Same!

    MIDNIGHT WRAP: min(diff, 1440-diff) ensures 11:30pm vs 12:30am = 60 min, not 1380 min.



    ml/allocator.py — OR-Tools CP-SAT Optimization


    def allocate_rooms(db, session_id):

    # Load: room_inventory {"2":20,"3":10}, students with traits, compatibility scores

    # Group students by (preferred_room_size, gender)

    # e.g. group ("2","Male") = all males who want double rooms


    for (room_size, gender), group in groups.items():

    assignments = _run_ortools(group, room_size, available_rooms, scores)

    # Save to recommendations + recommendation_members tables


    Inside _run_ortools (the CP-SAT solver):

    model = cp_model.CpModel()


    # VARIABLES: x[student][room] = 1 if student is in that room, else 0

    x = {(s,r): model.NewBoolVar() for s in students for r in rooms}


    # CONSTRAINT 1: Each student in exactly ONE room

    for s: model.AddExactlyOne([x[(s,r)] for r in rooms])


    # CONSTRAINT 2: No room over capacity

    for r: model.Add(sum(x[(s,r)] for s) <= capacity)


    # OBJECTIVE: Maximize total pairwise compatibility

    # both_in_r = 1 ONLY IF both students i and j are in same room r

    model.Maximize(sum(score[i][j] * both_in_r for all pairs))


    solver.Solve(model) # OR-Tools finds the global optimum


    CONSTRAINT PROGRAMMING: You define constraints + objective. Solver finds optimal solution.

    You don't write the search algorithm — OR-Tools does it.

    HARD CONSTRAINTS happen BEFORE the solver: gender groups + room size groups.

    11. Scoring Algorithm — Worked Example


    Student A: noise=0.8, clean=0.9, social=0.3, sleep="23:00", wake="07:00", study="silent"

    Student B: noise=0.7, clean=0.8, social=0.4, sleep="23:30", wake="07:30", study="quiet"


    Start: score = 100.0


    Noise: |0.8 - 0.7| = 0.1 × 20 = 2.0 → 98.0

    Clean: |0.9 - 0.8| = 0.1 × 20 = 2.0 → 96.0

    Social: |0.3 - 0.4| = 0.1 × 10 = 1.0 → 95.0

    Sleep: |1380-1410| = 30min → 30/180=0.167 → 0.167×15=2.5 → 92.5

    Wake: |420-450| = 30min → 2.5 → 90.0

    Study: both "silent" cluster → 0 penalty → 90.0


    RESULT: 90.0% compatibility (great match!)



    12. Optimization Algorithm — Worked Example


    4 students, 2 double rooms, all same gender:

    Alice-Bob=90, Alice-Carol=60, Alice-Dave=70, Bob-Carol=80, Bob-Dave=50, Carol-Dave=85


    Possible assignments:

    Option 1: {Alice,Bob} + {Carol,Dave} → 90+85=175 ← OR-Tools picks this!

    Option 2: {Alice,Carol} + {Bob,Dave} → 60+50=110

    Option 3: {Alice,Dave} + {Bob,Carol} → 70+80=150


    OR-Tools guarantees the GLOBAL OPTIMUM (175), not just a locally good result.

    Greedy algorithms can get stuck (e.g. pick Alice-Bob=90, then be stuck with 50+70=120).


    Hard constraints (before solver runs):

    1. preferred_room_size: student groups separate by desired room size

    2. gender: males and females never in the same group
